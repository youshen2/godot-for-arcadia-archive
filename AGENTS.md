AGENTS.md

本项目是一个基于 Godot Engine 的引擎/模块级项目。
AI Agent 在修改本仓库代码时，请严格遵守以下约定。

基本要求

* 修改前先理解当前代码结构，不要盲目重构。
* 尽量保持 Godot 原有代码风格、命名风格和目录结构。
* 不要引入不必要的大规模架构变化。
* 不要删除现有功能、接口或兼容逻辑，除非任务明确要求。
* 涉及引擎 API、类注册、绑定、文档生成的修改时，需要同步检查相关构建链路。

新增功能要求

当新增任何面向脚本层使用的功能时，必须同时考虑以下两类脚本接口：

1. GDScript 接口
2. Mono / C# 接口

也就是说，如果新增了某个类、方法、属性、枚举、信号或常量，并且它应当暴露给 Godot 脚本层使用，那么需要确保它不仅能在 GDScript 中正常调用，也要能在 Mono / C# 中正常使用。

不要只实现 GDScript 可见接口而忽略 Mono 绑定。

接口暴露规范

新增或修改脚本 API 时，请检查并维护：

* ClassDB 注册
* 方法绑定
* 属性绑定
* 枚举绑定
* 常量绑定
* 信号绑定
* 默认参数
* 参数名
* 返回值类型
* Mono / C# 绑定兼容性

如果接口会影响 Godot 的 API 文档生成，也需要同步更新文档。

doc_class 文档要求

所有新增的脚本可见类、方法、属性、枚举、信号、常量等，都需要补充对应的 doc_class 文档。

文档要求：

* 必须使用中文编写。
* 描述要清晰说明用途、参数含义、返回值含义和使用注意事项。
* 不要只写一句过于笼统的说明。
* 如果功能有边界条件、平台限制或性能注意事项，需要写入文档。
* 如果新增 API 同时面向 GDScript 和 Mono / C# 使用，文档中应避免只使用某一种语言的表达方式。

示例要求：

<method name="example_method">
    <return type="bool" />
    <param index="0" name="path" type="String" />
    <description>
        根据指定路径执行示例操作。返回 [code]true[/code] 表示操作成功，返回 [code]false[/code] 表示路径无效或操作失败。
    </description>
</method>

Mono / C# 兼容性要求

涉及 Mono / C# 的接口时，请特别注意：

* C# 侧命名是否符合 Godot 的绑定规则。
* 类型是否可以正确映射到 C#。
* 默认值是否能正确生成。
* 枚举、Flags、Variant、Array、Dictionary、Callable、Signal 等类型是否兼容。
* 不要使用只能被 GDScript 正常识别、但 C# 无法可靠绑定的接口设计。
* 修改底层绑定后，需要考虑是否影响生成的 C# API。

如果某个功能暂时无法提供 Mono / C# 接口，必须在提交说明或代码注释中明确说明原因。

文档语言要求

本项目的新增 doc_class 内容统一使用中文。
不要新增英文 doc_class 描述，除非修改的是已有英文文档并且任务明确要求保持英文。

中文文档应尽量自然、准确，避免机器翻译腔。

构建与验证

完成修改后，应尽可能进行以下检查：

* 确认项目可以正常编译。
* 确认新增 API 能在 GDScript 中调用。
* 确认新增 API 能在 Mono / C# 中调用。
* 确认文档生成不会报错。
* 确认没有破坏已有测试或已有 API。

如果无法完整执行构建或测试，需要在最终回复中说明哪些步骤未执行，以及原因。

修改原则

优先选择最小、清晰、可维护的修改方案。

不要为了完成单个功能引入过度抽象。
不要改动无关文件。
不要格式化整个仓库。
不要随意调整已有公共 API 的行为。

最终回复要求

完成任务后，请在回复中说明：

* 修改了哪些文件。
* 新增或调整了哪些功能。
* 是否同步提供了 GDScript 和 Mono / C# 接口。
* 是否补充了中文 doc_class。
* 执行了哪些构建或测试。
* 有哪些未完成或需要人工确认的地方。
